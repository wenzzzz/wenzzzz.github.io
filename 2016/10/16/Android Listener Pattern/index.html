Android 实现 listener： Observer pattern (观察者模式)

发表于 2016-10-16   |   分类于 Android

在Android中写界面的常常要用到listener，设计实现listener又常常要用到设计模式中的观察者模式，也就是observer pattern. 下面我们简单理解一下这个设计模式是怎么回事。

先来看什么是listener，监听器要实现的是，当一个Object发生了某些改变的时候，我们需要某些其他的Objects也做一定响应。那个最初发生变化的就是被观察者，那些需要监听被观察者变化的叫观察者，当被观察者变化时，那些观察者就需要做相应变化。这就是观察者模式了。

具体来看一个例子。假如我们有一个edittext输入框，在这个输入框输入结束的时候，我们想让别的几个textview都显示用户输入的text，同时数据库在这个时候进行插入操作，并且当前activity结束并start另一个activity。这个时候我们就需要这个模式。在这个例子里，那个edittext就是被观察者(Subject), 其余出现的类，比如那几个textview, 数据库类和当前的activity都属于观察者(observer)，因为它们都需要在被观察者变化的时候也做出相应的改变。

那么一般这种关系的代码要如何设计呢？我们这里把被观察者类叫Subject, 观察者类叫Observer, 那么Subject里肯定要维护一个list，这个list里存的就是所有的Observer，这样当Subject发生某些变化（调用自身某个方法）的时候，才能知道要去通知谁。而这些Observer肯定要都有一个同名的函数来应对Subject的这个变化，否则Subject也不知道怎么去通知Observer（调用Observer的哪个方法）。所以最好的方法就是Observer们都去实现一个interface，并实现interface里的方法。这样Subject也就不用管Observer具体都是什么类了，直接以那个Interface代替。

我们来看一个具体的简单代码例子吧，比如我们的被观察者是一个EditText，类名叫MyEditTextSubject，我们的观察者有一个TextView和一个ImageView，分别叫MyTextViewObserver和MyImageViewObserver。我们想做的是用户在MyEditTextSubject上输入完成按下回车键的时候，MyTextViewObserver显示用户输入，MyImageViewObserver根据用户输入显示一个图片。我们需要至少定义四个类，除了上面的三个类还有一个Interface，需要MyTextViewObserver和MyImageViewObserver来实现。

我们从外面往里面看吧。先想想在Activity里要怎么写。首先Activity的layout里要设置好这些类的位置和大小等信息，这部分代码我就不贴了，然后要在Activity里findViewById找到这三个类，然后把MyTextViewObserver和MyImageViewObserver加入到MyEditTextSubject的listener list里面。至于如何触发这个事件我们在MyEditTextSubject里面写。
public class HomePageActivity extends Activity {
@Override
public void onCreate(Bundle savedInstanceState) {
super.onCreate(savedInstanceState);
setContentView(R.layout.homepage); 

   MyEditTextSubject editText = (MyEditTextSubject) findViewById(R.id.my_edit_text);   
   MyTextViewObserver textView = (MyTextViewObserver) findViewById(R.id.my_text_view);   
   MyImageViewObserver imageView = (MyImageViewObserver) findViewById(R.id.my_image_view);   

   editText.addListener(textView);   
   editText.addListener(imageView);   
}

}
简单吧，这样我们就把Observer们都放到了Subject的list里，具体我们想监听什么和做什么就可以在各自的类里面实现了，外面的Activity就不管这些了。

接下来我们看看一个典型的被监听者Subject类都要定义哪些东西:
public class MyEditTextSubject extends EditText{
List listeners = new ArrayList();
public MyEditTextSubject(Context context, AttributeSet attrs) {
super(context, attrs);
setOnEditorActionListener(new EditText.OnEditorActionListener() {
@Override
public boolean onEditorAction(TextView v, int actionId, KeyEvent event) {
if (actionId == EditorInfo.IME_ACTION_SEARCH ||
actionId == EditorInfo.IME_ACTION_DONE ||
event.getAction() == KeyEvent.ACTION_DOWN &&
event.getKeyCode() == KeyEvent.KEYCODE_ENTER) {
// 就在这个时候触发各个Observer，这里抽象成了统一的listener interface
for(MyListener listener: listeners){
listener.update(v.getText().toString());
}
}
return false; // pass on to other listeners.
}
});
}
public void addListener(MyListener listener){
listeners.add(listener);
}
public void removeListener(MyListener listener){
listeners.remove(listener);
}
}
在Subject类里一般要有addListener和removeListener，也可以把遍历listener的代码写在函数里。我们上面的代码就是在这个edittext输入完成点击回车的时候，把edittext里的string传给所有list里的Observer，通过他们内部定义的update()这个函数。

我们先来看Observer们都要实现的这个interface吧，我们这里只要求这些observer实现一个函数就够了，那就是update()。update()里的输入其实可以是任何东西，包括可以把Subject本身传过去，这里我们只需要MyEditTextSubject里当前的输入所以就只传这个string值。
public interface MyListener {
public void update(String content);
}
好了，现在我们就是让我们的Observer来实现这个update()方法就好了。
public class MyTextViewObserver extends TextView implements MyListener{
public MyTextViewObserver(Context context, AttributeSet attrs) {
super(context, attrs);
}
@Override
public void update(String content) {
this.setText(content);
}
}
上面是我们的textview的observer，可以看到它implements MyListener，并且实现了update()，具体就是把自己设成传进来的string。
public class MyImageViewObserver extends ImageView implements MyListener{
public MyImageViewObserver(Context context, AttributeSet attrs) {
super(context, attrs);
}
@Override
public void update(String content) {
if(content.equals(“Bird”)){
setImageResource(R.drawable.bird);
}
else{
setImageResource(R.drawable.lion);
}
}
}
上面是我们的imageview的observer，它同样实现了update()方法，可以看到如果传进来的string是“Bird”的话它就显示一个叫bird的drawable图片，否则显示另外一张。

以上这些代码虽然简单，但也组成了一套观察者设计模式，在工程中如果我们要实现这种listener机制的话一般还是要用这种设计模式。它的本质是当修改目标对象的状态的时候，就会触发相应的多个观察者，当要实现一对多的关系的时候也要想到这个模式。而且这个过程还可以是动态的，因为可以addListener和removeListener来动态选择观察者。希望本文能起到一个抛砖引玉的作用。
