Android ListView 介绍和 Holder 设计模式

发表于 2016-10-16   |   分类于 Android   |  

在Android中ListView是一个非常常见的类，但是它的用法又和普通的textview或imageview不太一样，因为它继承了AdapterView，是从adapter获取数据来显示的，显示的时候又用到Holder的设计模式来提高性能。但掌握起来只要转换一下思路就好了，而且这种思路还经常被用到，稍微复杂点的list结构都会用到这种思路。

  1. 我们先来看一下ListView的基本原理吧。ListView可以以列表的方式来显示用户的数据。它继承于AdapterView, 我们可以用M(Model)V(View)C(Controller)的模式来理解listview，Model是数据来源，可以是从数据库读取，也可以接收网上传送来的数据，最简单的例子就是一个hard coded的数组。Controller就是ListView中的Adapter，Adapter就是一个数据和View之间的中介，负责怎么处理这些数据以及具体显示在View的哪些地方、怎么显示等问题，Android中有专门的adapter类，我们只要继承这个类，覆盖实现某些函数，
    然后在listview中运行setAdapter()就能完成大部分我们需要的功能。View就是ListView，来定义这个view的大小位置等信息。

  2. 理解listview，重点是理解它的adapter。在android中，Adapter类是AdapterView(ListView的父类)和数据之间的桥梁。这个类常见的adapter有ArrayAdapter, SimpleAdapter等，对应于不同的数据传入方式。我们这里选择最简单的ArrayAdapter来用一个例子看一下这个Adapter类是如何充当View和数据之间桥梁的。

比如我们想实现一个listview，这个list里是各种不同的菜谱，为了简单，这些菜谱数据就存在代码数组里。我们先来看看用到这个listview的activity是什么样子的，我们这里用fragment来写，道理一样。
public class CookBookFragment extends Fragment {
ListView recipeList;
public CookBookFragment() {
}
@Override
public View onCreateView(LayoutInflater inflater, ViewGroup container, Bundle savedInstanceState) {
View v = inflater.inflate(R.layout.cookbook, container, false);
recipeList = (ListView) v.findViewById(R.id.recipe_list);
RecipeItem[] recipesData = new RecipeItem[1];
recipesData[0] = new RecipeItem(“Hello Title”, “Hello Content”);
RecipeListAdapter adapter = new RecipeListAdapter(container.getContext(), R.layout.recipe_item, recipesData);
recipeList.setAdapter(adapter);
return v;
}
}
在cookbook.xml里，我们只写了一个叫recipe_list的listview。可以看到上面这段代码只做了一件事，就是给这个listview setAdapter。重点就是这个adapter了，初始化的时候我们传入了一个layout和一个数组，这个layout就是list中每一行的layout，这个数组就是我们要的数据，这个初始化方式是ArrayAdapter的初始化方式。下面我们来看看这个RecipeListAdapter是怎么写的。
public class RecipeListAdapter extends ArrayAdapter{
Context context;
int layoutResourceId;
RecipeItem[] recipesData;
public RecipeListAdapter(Context context, int layoutResourceId, RecipeItem[] recipesData) {
super(context, layoutResourceId);
this.context = context;
this.layoutResourceId = layoutResourceId;
this.recipesData = recipesData;
}
@Override
public int getCount() {
return recipesData.length;
}
@Override
public View getView(int position, View convertView, ViewGroup parent) {
View row = convertView;
RecipeItemHolder holder = null;
if(row == null) {
LayoutInflater inflater = ((Activity)context).getLayoutInflater();
row = inflater.inflate(layoutResourceId, parent, false);
holder = new RecipeItemHolder();
holder.title = (TextView)row.findViewById(R.id.recipe_title);
holder.content = (TextView)row.findViewById(R.id.recipe_content);
row.setTag(holder);
}
else {
holder = (RecipeItemHolder)row.getTag();
}
RecipeItem recipe = recipesData[position];
holder.title.setText(recipe.title);
holder.content.setText(recipe.content);
return row;
}
static class RecipeItemHolder {
TextView title;
TextView content;
}
}
上面这段代码里，除了构造函数，override了两个函数，另外定义了一个静态变量。构造函数把传进来的类存为成员变量，方便其他函数引用。override的两个函数，是ArrayAdapter必须知道的两个函数，一个是getCount()，告诉这个listview一共要显示几个item。 然后是geiView(int position, …), 返回的是在listview上position这个位置我们要显示的View。也就是说在listview中每个位置的item可以分别单独定义view，通过getView来实现这一点，它return的那个view就是在position位置显示的view，具体解释我们下面讲，那个静态类是为了getView()中的Holder设计模式服务的，我们也在下面讲。

  1. 让我们再来看getView的代码。
    @Override
    public View getView(int position, View convertView, ViewGroup parent) { 
    ...
    }
    如果我们的item很多，那么android是否会真的建立这么多的view，当view在屏幕位置的时候就显示，不在的时候就存在内存里面呢？答案是否定的，因为这样太浪费内存了。Android内部用了回收机制，将没有在当前屏幕上显示的item放进RecycleBin，如果滑动到应该显示该item时再从RecycleBin中复用这个View。例如，屏幕上只能够显示5个item，我们有6个item在这个list上，那么当我们向下滑把第一个滑出屏幕，第6个要出现时，系统就从RecycleBin中重用第一个view，只不过重新设置为第6个item的数据，而不会去重新创建一个view。

明白了这个原理后，我们来看传入的三个参数。第一个是要显示view的position，如果我们的listview里一共有10个item的话，那么这个getView就会调用10次，position从0到9。第二个参数convertView就是刚刚说的系统回收利用的view，如果它是null的时候我们可以新建，如果它不是null说明就是系统回收后的view，我们可以重新设置它而不必重新创建。第三个参数是parent view，调用inflater.inflate()的时候可作为参数传进去（见上面getView()内具体代码）。

好了，现在不明白的可能就是getView里Holder模式的运用。大家可能知道，在android编程中，findViewById()这种操作是很消耗时间的，而getView()会被非常频繁的调用到，所以我们不希望每次调用getView()时都运行findViewById()来找到相应的view。根据listview的recycle原理，绝大部分view都是被回收利用的，所以我们完全可以在view里保存对其child view的引用，这样我们就不用每次都findViewById来找其child view了。

Android里每个view都可以通过setTag(Object tag)来保存一个Object到view本身，我们可以每次新建view的时候把上面的child view通过findViewById找到，然后把它们的reference存到这个view的tag里，比如我们上面的代码，每个item的view里都有两个textView，我们就可以只在新建item view的时候通过findViewById来找到这两个textView，然后存到这个item的tag里。然后当这个view再次被利用的时候，我们就不必再调用findViewById了，直接调用getTag来得到两个textView的reference。大家可以再看看上面getView()的代码来加深理解。这个存child view并传入setTag()的Object就成为Holder. Holder模式就是通过这种方式来提高app的运行效率。
